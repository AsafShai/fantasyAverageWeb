Live Player Projections — Implementation Plan

Supersedes plans/live_daily_player_projections.md — that draft assumed no live pipeline existed yet. It now does: the nightly job (app/services/model_nightly_service.py) already writes real per-night results and materializes feature vectors in Postgres, and ModelNightlyService.get_inference_store() already exposes a live, DB-backed FeatureStore — currently unused by anything except the nightly job itself. This plan wires that existing store to two UI surfaces instead of building new DB tables / a new daily job.

1. Decisions (from planning session)

Live nightly store only — no fallback to the historical replay/simulation store. If there are no games today, say so; don't fake it with replay data.
Scope: today's games only, no history browsing.
Player universe = same as Matchup today (ESPN roster + FA + waivers, matched to the model store by player name — same informal join convention MatchupCell already uses).
One shared backend service (LiveProjectionService) powers both surfaces — no duplicated "today's games + live store" logic.
Two surfaces: (1) a new standalone /projections page, (2) folded into the existing Matchup expand row (Players.tsx + TeamDetail.tsx) — not a third separate table column.
Stat set = the league's 8 roto categories (PTS, REB, AST, 3PM, STL, BLK, FG%, FT%), matching every other table in the app. Shooting volume (FGM/FGA, FTM/FTA) shown small, stacked under the FG%/FT% number — no separate columns — reusing Simulation's VFrac pattern exactly.
Minutes: default (recent-avg) projection shown immediately, plus a slider to re-predict at custom minutes — in both surfaces, for now. If Matchup's expand row gets too cluttered, drop the slider there later and keep it only on the dedicated page.
No visible confidence range (low/high numbers) — too noisy for a casual glance.
Integer-vs-decimal display is a user toggle, not a fixed choice — reuse Simulation's integerMode checkbox pattern (fmt(), shootingCell(), ptsIntFromComponents()) in both surfaces. Off by default matches Simulation's default (checked/int on) — decide at build time which default reads better for a projections context.
Confidence instead shown as a 3-tier colored dot (green/amber/red), reusing Simulation's existing status + reason semantics. Insufficient history (red) shows small gray text ("not enough data") instead of numbers.
Performance: singleton LiveInference (models loaded once, not per request), batch predict_many() for the default line embedded in /matchups/today, single debounced predict() call only when the slider moves.

2. Data Flow

Request-time flow
ESPN scoreboard (today's games) ──┐
                   ├──▶ LiveProjectionService (singleton)
ModelNightlyService.get_inference_store() ──┘        ├─ warm LiveInference (models loaded once)
                                     ├─ predict_many() → batch default lines
                                     └─ predict() → single-player custom-minutes

GET /api/matchups/today ──▶ enriched PlayerMatchupResponse (matchup + projection fields)
        ├──▶ TeamDetail.tsx (Matchup expand row)
        ├──▶ Players.tsx (Matchup expand row)
        └──▶ Projections.tsx (new page — same objects, different rendering)

POST /api/projections/predict {player_name, minutes} ──▶ single re-predicted line (slider only)

6. Performance Notes

  • LiveInference.__init__ loads all .joblib models from disk — happens once, on the singleton's first construction, not per request.
  • get_inference_store() is already cached on ModelNightlyService and only invalidated by the nightly job — no rebuild mid-day.
  • Default lines for the whole slate come from one predict_many() batch call per /matchups/today request (vectorized — one DataFrame + one model.predict() per stat, not per player).
  • The slider only triggers predict() (single player) on release/debounce (350ms), matching Simulation.tsx's existing pattern — no request storm while dragging.

7. Open Risks

  • Name-matching ESPN players to the store's PLAYER_NAME is a string join, not an ID join — Jr./Sr./accent mismatches will silently drop a player's projection. Same risk already exists for Matchup today; not new, but not fixed either.
  • Off-season / no-games-today: page should render an honest "no games today" state, not an empty table with no explanation.
  • Slider-in-both-places was agreed as "try it, remove later if cluttered" — expect a follow-up pass to drop it from the Matchup expand row if it proves too dense on mobile.
